Knowledge Routing

When something is learned in a session, it needs to flow to the right place. This guide defines what goes where.

The Durability Test

Before routing any knowledge, ask yourself:

"If someone new joined CDE in 6 months and read this, would it help them understand how the business, client, or system works?"

YES = durable. Route it.    NO = skip it.
Route These (Durable)
How a business operates
How CDE solves problems
Strategic context (why X over Y)
Protocols and standards
Corrections to Claude's behavior
Infrastructure decisions
Skip These (Not Durable)
Action items and to-dos
Meeting schedules
Temporary state ("waiting on X")
Debugging steps
One-off Q&A
Status updates that go stale in days

The Four Pillars

All durable knowledge routes to one of these four pillars:

PillarWhat BelongsWhere It Lives
1. Knowledge Base Repeatable processes, standards, protocols, "how CDE does X" reference/ docs, toolkit pages, memory reference files
2. Client Playbooks Strategic client context — business model, audience, CDE approach Memory project files, client playbook repos
3. The CDE Way Feedback, corrections, quality standards, tool preferences Memory feedback files
4. Operational Docs Data agents depend on — staffing, contacts, scope, agent status Google Sheets (Client Tracker, Agent Registry)
Playbooks are strategic only. No tasks, action items, timelines, checklists, or meeting notes. If it reads like a task — skip it.

Decision Tree

Use this sequential filter to route any piece of knowledge:

1. Durable? (6-month test) → NO → Skip → YES → Continue 2. Changes Claude's behavior? → YES → Pillar 3 (feedback memory file) 3. Changes a repeatable process? → YES → Pillar 1 (reference doc + toolkit page) 4. Strategic client context? → YES → Pillar 2 (project memory + playbook) 5. Data agents depend on? → YES → Pillar 4 (operational sheets) 6. CDE infrastructure? → YES → Pillar 1 + 4 (toolkit + registry) 7. None of the above → Skip

Common Patterns

One piece of knowledge often flows to multiple places. These are the patterns:

Client Decision
1. Memory project file (context + rationale)
2. Client playbook (strategic implication)
3. Client Tracker sheet (scope change if any)
New Protocol
1. Reference doc (source of truth)
2. Memory reference file (index entry)
3. Toolkit page (team-facing version)
Feedback / Correction
1. Feedback memory file (rule + why + how)
2. Flag Master Brain Doc (if system-wide)
Infrastructure Change
1. Workspace memory (updated fact)
2. Toolkit page (setup/architecture docs)
3. Agent Registry (changelog entry)
