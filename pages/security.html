Security Policy

The do's and don'ts of working with AI tools at CDE.

The Rules

Never Do

✗Paste API keys, passwords, or tokens into a Claude conversation
✗Commit credentials to any git repository
✗Let Claude run Windows system commands (reg, dism, sfc, etc.) unsupervised
✗Disable or bypass security hooks
✗Force-push to main/master branches
✗Run rm -rf or mass deletion commands through Claude
✗Share your Claude credentials or MCP server tokens with anyone outside CDE

Always Do

✓Store credentials in the local credentials/ directory only
✓Review destructive commands before approving them
✓Keep hooks enabled and up to date
✓Use the allow/deny lists in settings.json to control what Claude can do
✓Ask before giving Claude access to new services or APIs
✓Report any unexpected behavior or security concerns to Balaji immediately

Permission Model

Claude Code asks permission before running commands. You control how strict it is:

ModeBehaviorRecommended For
DefaultAsks before every commandNew users, learning the system
Allow ListAuto-approves known-safe patterns (git, npm, etc.)Experienced users
Deny ListHard blocks on dangerous patterns regardless of user approvalEveryone (enforced by hooks)
Even in the most permissive mode, hooks still block dangerous operations. The deny list is a hard floor that cannot be bypassed from within Claude Code.

Incident Response

If something goes wrong:

  1. Stop Claude immediately — press Ctrl+C or close the session
  2. Assess the damage — check what commands ran, what files changed
  3. Notify Balaji — even if you think it's minor
  4. Don't try to fix it with Claude — if something is broken at the system level, fix it manually
